Booth-based events are broken by bad navigationLos eventos con stands sufren por una mala navegación

Booth-based events — career fairs, trade shows, exhibitions — rely on static maps or QR codes that can't update when layouts change. Attendees miss booths. Exhibitors lose visibility. A Capital One event coordinator told us they had stopped attending career fairs entirely because of low engagement. Event organizers have no real-time insight into foot traffic or attendee behavior.

Existing solutions (Navigne, Pointr, Esri, Atrius) are built for fixed environments — malls, hospitals, airports — not temporary, frequently-reconfigured event spaces. InMaps fills this gap.Los eventos con stands — ferias de empleo, ferias comerciales, exposiciones — dependen de mapas estáticos o códigos QR que no se actualizan cuando cambia la distribución. Los asistentes se pierden stands. Los expositores pierden visibilidad. Una coordinadora de Capital One nos contó que habían dejado de asistir a ferias de empleo por completo por la baja participación. Los organizadores no tienen visión en tiempo real del tráfico de asistentes.

Las soluciones existentes (Navigne, Pointr, Esri, Atrius) están diseñadas para entornos fijos — centros comerciales, hospitales, aeropuertos — no para espacios de eventos temporales. InMaps llena este vacío.

MOBILE APPAPP MÓVIL
Flutter — iOS and AndroidFlutter — iOS y Android

The app reads the CSV from the setup platform, converts it to a pixel matrix (1 = walkable, 0 = blocked), and runs A* Search to calculate the optimal path. As the user walks, BLE signals and IMU data are fused continuously: BLE provides absolute position, IMU provides step detection at 0.7m per step, and the phone compass determines facing direction.La app lee el CSV de la plataforma, lo convierte en una matriz de píxeles (1 = transitable, 0 = bloqueado) y ejecuta A* para calcular la ruta óptima. Al caminar, señales BLE e IMU se fusionan continuamente: BLE da posición absoluta, IMU detecta pasos a 0,7m por paso y la brújula determina la dirección.

When BLE signal is lost, IMU dead-reckoning maintains position for up to 30 seconds. When signal returns, position is re-anchored to BLE. Average latency: 0.87 seconds across three tested venues.Cuando se pierde la señal BLE, el dead-reckoning por IMU mantiene la posición hasta 30 segundos. BEACON PLACEMENTCOLOCACIÓN DE BALIZAS
Hexagonal grid algorithmAlgoritmo de cuadrícula hexagonal

The beacon placement algorithm takes venue width, height, and beacon range as inputs and generates a hexagonal grid of positions. Rows alternate with a dx/2 offset. A minimum of 3 beacons is guaranteed using an equilateral triangle fallback. Validated by simulation: hexagonal layout used 31.7% fewer beacons than random placement for equivalent coverage.El algoritmo toma el ancho, alto y rango de balizas como entrada y genera una cuadrícula hexagonal de posiciones. Las filas alternan con un desplazamiento de dx/2. Se garantiza un mínimo de 3 balizas con un triángulo equilátero de respaldo. Validado: el diseño hexagonal usó un 31,7% menos de balizas que la colocación aleatoria.

A* PATHFINDINGPATHFINDING A*
Optimal path with obstacle avoidanceRuta óptima con evitación de obstáculos

A* Search runs on a pixel matrix derived from the venue layout. Each cell is 0 (blocked — wall, table, booth) or 1 (walkable). The heuristic is Manhattan distance. When the user moves off-path, A* recalculates within 2 seconds. POST requests upload path updates to Firebase, which streams them to the app frontend.A* se ejecuta sobre una matriz de píxeles derivada del plano del local. Cada celda es 0 (bloqueada — pared, mesa, stand) o 1 (transitable). La heurística es la distancia Manhattan. Cuando el usuario se desvía, A* recalcula en 2 segundos.

DOE EXPERIMENTATIONEXPERIMENTACIÓN DOE
Full factorial design of experimentsDiseño factorial completo de experimentos

A 3×3×3 full factorial DOE tested beacon height (1.2m, 1.5m, 1.8m), spacing (15m, 20m, 30m), and zone type (corridor, hotspot, enclosed room) across 3 venues with 2 replications each — 54 test conditions total. Result: 1.5m height + 15m spacing achieved the lowest average error of 0.41m (±0.07m).Un DOE factorial 3×3×3 probó altura (1,2m, 1,5m, 1,8m), separación (15m, 20m, 30m) y tipo de zona en 3 sedes con 2 réplicas cada una — 54 condiciones de prueba. Resultado: 1,5m de altura + 15m de separación logró el error medio más bajo: 0,41m (±0,07m).

SENSOR FUSIONFUSIÓN DE SENSORES
BLE + IMU dead reckoningBLE + dead reckoning IMU

Position is maintained through continuous fusion of two signals. BLE RSSI gives absolute position with ~0.41m accuracy. The phone's IMU provides step detection at 0.7m per step with compass-based direction. When BLE is degraded, IMU dead-reckoning maintains continuity. Auto-reconnect restores BLE anchor within 3 seconds in 98%+ of handover events.La posición se mantiene fusionando continuamente dos señales. El BLE RSSI da posición absoluta con ~0,41m de precisión. El IMU del teléfono detecta pasos a 0,7m por paso con dirección por brújula. Cuando el BLE se pierde, el dead-reckoning IMU mantiene la continuidad.
